uvlink — One MicroVM, Many Apps

Design spec for reaching several apps running inside a single Lambda MicroVM through one uvlink process. Scoped deliberately to this one topology — no fan-out, no fan-in, no fleet.

01 The problem

You snapshot one MicroVM image that starts several services. uvlink today reaches exactly one of them.

A single dev/sandbox VM commonly runs more than one thing on different in-VM ports:

JupyterLab :8888 TensorBoard :6006 Vite / Next dev :3000 an API :8080 code-server :8443

Right now you'd launch four uvlink processes on four local ports and juggle them. The goal: one uvlink, one command, all the apps on that VM reachable from the browser.

Why this is the right thing to build first. It's the most common real setup, and it needs no fleet logic, discovery, stickiness, or load balancing — those only appear once there's more than one VM. Multi-app-on-one-VM is purely a question of which in-VM port does this request go to.

02 The mechanic uvlink rests on

The in-VM destination port is chosen per request, by uvlink — not fixed by the VM.

Every MicroVM request carries the target in-VM port alongside the auth token. uvlink sets both:

ProtocolAuth carried byIn-VM port carried by
HTTP X-aws-proxy-auth: <token> X-aws-proxy-port: 8888
WebSocket lambda-microvms.authentication.<token> subprotocol lambda-microvms.port.8888 subprotocol
Because uvlink already writes X-aws-proxy-port / lambda-microvms.port.N on every request, routing to multiple apps is just choosing that number per request. The token only has to permit the port (see §6). No platform change, no new protocol — a small, local decision.

03 What uvlink does today (and the one change)

Today backendPort is a single constant baked in at createProxy time.

Today

createProxyServer({
  endpoint, getToken,
  backendPort: '8080'   // one fixed port
});

Every request — HTTP and WS — gets the same hardcoded port. One app only.

The change

createProxyServer({
  endpoint, getToken,
  resolvePort(req)      // pick port per request
});

Replace the constant with a small function of the request. Single-app mode becomes a function that returns a constant — fully backwards compatible.

That's the whole core change: backendPort → resolvePort(req). Both the HTTP handler and the WS upgrade handler call it. Everything in §4 is just different resolvePort implementations. As a bonus this is the same shape that lets us delete the drifted duplicate proxy in pool.mjs later.

04 Two ways to map browser → in-VM ports

For one VM with many apps there are two practical mappings. They differ in robustness.

Mode A — Listener-per-port recommended default

uvlink binds several local ports; each maps to one in-VM port.

localhost:3000  ──▶  VM :8888  (Jupyter)
localhost:3001  ──▶  VM :6006  (TensorBoard)
localhost:3002  ──▶  VM :3000  (dev server)

zero rewritingevery app sees itself at root

Each app gets its own clean origin. Nothing touches paths, redirects, or cookies — so even the pickiest SPA works. resolvePort = "the port bound to the listener that received this request."

Cost: N local ports to remember and (if remote) tunnel.

Mode B — Path-prefix opt-in

One local port; a path prefix selects the app.

localhost:3000/jupyter/*  ──▶  VM :8888
localhost:3000/tb/*       ──▶  VM :6006
localhost:3000/api/*      ──▶  VM :8080

single originone port to tunnel

⚠ The app must tolerate a base path. Jupyter (--ServerApp.base_url) and TensorBoard (--path_prefix) do; many SPAs don't. Needs Location-header rewriting and care with cookie Path. This is where "works for Jupyter, breaks for my SPA" reports come from.

Recommendation: ship Mode A first — it's robust and covers the common case with no rewriting. Add Mode B as an explicit opt-in for people who specifically want a single shareable origin and run base-path-friendly apps. (Subdomain routing — jupyter.localhost — is deferred; it needs DNS setup and adds little for a single VM.)

05 Request flow, end to end

What actually happens to one request, for both protocols, in Mode A.

HTTP

1
Browser hits localhost:3001/ (the TensorBoard listener).
2
resolvePort(req) → 6006 (this listener's mapped in-VM port).
3
Token provider returns the current (cached, auto-refreshed) JWE.
4
uvlink opens HTTPS to the VM endpoint with headers X-aws-proxy-auth: <jwe> + X-aws-proxy-port: 6006 and pipes the response back.

WebSocket (e.g. a Jupyter kernel on the :8888 listener)

1
Browser upgrades on localhost:3000/api/kernels/…/channels.
2
resolvePort(req) → 8888.
3
uvlink builds Sec-WebSocket-Protocol: lambda-microvms, lambda-microvms.authentication.<jwe>, lambda-microvms.port.8888, + the app's own subprotocol (v1.kernel.websocket.jupyter.org) preserved.
4
TLS-connects to the VM, replays the upgrade, then pipes bytes both ways.
The WS upgrade is a separate handler from HTTP in the current code. Both must call the same resolvePort, or a kernel socket lands on the wrong app's port. This is the single easiest thing to get wrong — call it out in tests.

06 One token, scoped to the apps you expose

A single MicroVM means a single token — but it should permit exactly the ports you route to.

Today uvlink mints with allowedPorts = [{"allPorts":{}}] — the token can reach any port in the VM. For multi-app it's cleaner and safer to scope it to just the apps you expose:

aws lambda-microvms create-microvm-auth-token \
  --microvm-identifier microvm-… \
  --expiration-in-minutes 60 \
  --allowed-ports '[{"port":8888},{"port":6006},{"port":8080}]'

One token covers all apps

Same VM, same auth boundary — no need for a token per app. The existing cache + auto-refresh provider is unchanged; it just mints a wider-scoped token.

Least privilege is the only real choice

If you expose uvlink beyond localhost, allPorts means a reachable proxy can hit internal ports you never meant to publish (a debug port, a DB on :5432). Scoping to the routed ports removes that.

Derive allowedPorts automatically from the configured routes (the set of in-VM ports), with an override flag. The user states the apps once; the token scope follows.

08 API & CLI sketch

CLI (Mode A — the headline ergonomics)

uvlink --microvm-id microvm-… --region us-east-1 \
       --map 3000:8888 \      # Jupyter
       --map 3001:6006 \      # TensorBoard
       --map 3002:8080        # API

  ✓ Jupyter      http://localhost:3000
  ✓ TensorBoard  http://localhost:3001
  ✓ API          http://localhost:3002
  (token scoped to ports 8888, 6006, 8080; auto-refreshing)

CLI (Mode B — single origin, opt-in)

uvlink --microvm-id microvm-… --region us-east-1 --port 3000 \
       --route /jupyter=8888 \
       --route /tb=6006 --strip-prefix

Library

// Backwards compatible — single app, unchanged:
createProxy({ endpoint, microvmId, region, port: 3000, backendPort: 8888 });

// Mode A — listener-per-port:
createProxy({
  endpoint, microvmId, region,
  routes: [
    { listen: 3000, backendPort: 8888 },
    { listen: 3001, backendPort: 6006 },
    { listen: 3002, backendPort: 8080 },
  ],
  // allowedPorts derived from routes; override with allowedPorts: [...]
});

// Mode B — path-prefix on one origin:
createProxy({
  endpoint, microvmId, region, port: 3000,
  routes: [
    { prefix: '/jupyter', backendPort: 8888, stripPrefix: true },
    { prefix: '/tb',      backendPort: 6006, stripPrefix: true },
  ],
});
Open questions: (a) --map a:b vs --route /p=port naming; (b) should a single shared local port with auto-assigned listeners be possible, or always explicit; (c) default stripPrefix on or off in Mode B; (d) print a readiness check per app at startup, or lazily on first request.

09 Build plan

Small, ordered, each step shippable.

1
Core: backendPort → resolvePort(req) in proxy.mjs, called from both the HTTP and WS handlers. Single-app path becomes a constant resolver — no behavior change, fully compatible.
2
Mode A (listener-per-port). createProxy({ routes:[{listen,backendPort}] }) binds one server per listener sharing the token provider. CLI --map local:vm. The common case, no rewriting.
3
Scoped token. Derive allowedPorts from the route set; pass through mintViaCli. Flag to override.
4
Mode B (path-prefix), opt-in. Prefix match in resolvePort + stripPrefix + Location rewriting. Same logic in the WS upgrade path.
5
Tests + examples. Extend the mock backend to two ports; assert each request reaches the right one (HTTP and WS). Add a multi-app.mjs example.
First slice = steps 1–3. Delivers "one uvlink, all the apps on my VM" with least-privilege tokens and no rewriting risk. Mode B is a clean follow-up on the same resolvePort seam.
